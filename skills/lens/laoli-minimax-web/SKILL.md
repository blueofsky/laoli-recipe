# MiniMax 网页版 TTS 使用经验 Skill

## 定位

经验层技能，记录MiniMax网页版（platform.minimaxi.com）的TTS使用经验和最佳实践。不替代laoli-tts CLI工具，而是提供手动操作场景下的操作指南。

## 工具对比

| 维度 | laoli-tts CLI (minimax) | MiniMax网页版 |
|------|------------------------|--------------|
| 费用 | API按量计费 | 有免费积分（10000） |
| 操作 | 命令行，适合批量 | 浏览器界面，适合逐段调试 |
| 实时试听 | 需生成后播放 | 实时预览 |
| 参数控制 | 数值精确 | 滑块+预设 |
| 适用场景 | 批量自动化 | 创意调试、逐段确认 |

## 音色选择

|--------|------|------|---------|

### 不推荐

- ❌ 抒情男声/电台男主播——太"文艺"

## 参数设置

### 默认参数（适用于大部分场景）

| 参数 | 建议值 | 说明 |
|------|--------|------|
| 模型 | speech-2.8-hd | 最新模型，音质最好 |
| 语速 | **1（默认）** | 不需要刻意放慢，默认语速就是对的 |
| 声调 | **0（全程不变）** | 同一个人在叙述，声调变化会让听众觉得换了人 |
| 音量 | 2-3 | 稍大一点，但不要播音腔 |

### 关键原则

1. **声调全程不变（最重要）**：同一个人在叙述，不能因为场景不同就变声调。情绪变化通过语速和停顿体现，不是通过声调。如果每个场景都改声调，听众会觉得换了一个人在说话。
2. **语速用默认值（1）**：不需要刻意放慢。TTS引擎本身节奏就合适，慢了反而显得拖沓。协和号视频全部用语速1，效果很好。
3. **音量微调即可**：从2到3的变化就够了，不要差异太大。
4. **每个场景的参数尽量一致**：保持"同一个人在讲故事"的感觉。只在极端场景（如最沉重的一刻）做微小调整。

## 停顿标签语法

MiniMax的停顿标签格式：`<#秒数#>`

| 标签 | 含义 | 使用场景 |
|------|------|---------|
| `<#0.5#>` | 停顿0.5秒 | 短暂分隔 |
| `<#1#>` | 停顿1秒 | 句间自然停顿 |
| `<#1.5#>` | 停顿1.5秒 | 强调前的蓄势 |
| `<#2#>` | 停顿2秒 | 重要转折前 |
| `<#3#>` | 停顿3秒 | 长沉默，让画面说话 |

### 使用原则

**少即是多**。TTS引擎本身会在句号、逗号处自然停顿，不需要手动标每个停顿。

**只在需要"刻意沉默"的地方加标签**：
- ✅ 场景转折前："停不下来，<#1#>只能飞。"
- ✅ 长沉默开场："<#3#>一百一十三人，全部遇难。"
- ✅ 金句蓄势："终结一个时代的，<#1.5#>不是更大的飞机。"
- ❌ 每个逗号都加：会导致"一顿一顿"的机械感（协和号项目中踩过这个坑）

**全片停顿标签数量建议**：3-5个足够。协和号视频全片7段配音只用了3个停顿标签，效果很好。

## 长篇音频内容（视频号音频/播客）

当源文本超过1万中文字（约40分钟）时，不适合一次性录制，需要分集处理。

### 分集规划原则

1. **时长目标**：每集15-25分钟（约3750-6250中文字）。视频号音频格式支持长内容，但过长影响完播率
2. **按章节自然分组**：源文本通常有章节结构，按章节内容关联性分组，不要硬切
3. **平衡各集时长**：允许±30%浮动，但避免出现3分钟的超短集
4. **每集需要独立开头和结尾**：
   - 开头：简短回顾上集 + 引出本集主题（2-3句）
   - 结尾：总结本集要点 + 预告下集（2-3句）

### 文件输出规范

每个项目输出两类文件：
- `EP0X-标题.md` — 完整配音稿（含MiniMax参数、BGM关键词、停顿标签）
- `EP0X-标题-纯文本.txt` — 纯配音文本，去掉所有markdown头，直接复制粘贴到MiniMax

另外生成一个 `系列概览.md` 记录全系列分集结构、总时长、音色参数。

### 字数估算方法

中文TTS语速约250字/分钟（默认语速1）。估算公式：
```
预估时长(分钟) = 中文字数 / 250
```
⚠️ 这只是估算，实际录制后必须用 ffprobe 验证。

### 英文文本处理

源文本中常见英文人名注释（如 `Homo erectus`、`Gilgamesh`）。MiniMax中文TTS对英文的处理不稳定——可能朗读英文、跳过或发音怪异。建议：
- 保留英文注释，让用户自行决定是否删除
- 在纯文本文件中明确提示此问题

### MiniMax免费积分限制

MiniMax网页版有10000免费积分。长篇内容（如7集×20分钟）可能超出免费额度。建议：
- 先录制前1-2集测试效果
- 确认音色和参数后再批量录制
- 积分不足时考虑用 laoli-tts CLI（API按量计费）

## TTS 文稿预处理

录制前必须对源文本做预处理，否则 TTS 朗读效果会大打折扣。

### 括号内容处理（最重要）

括号中的内容（中英文括号均包含）在 TTS 朗读时会产生**割裂感**——引擎会在括号前后插入不自然的停顿和语调变化。

**处理规则：直接删除所有括号及括号内的内容。** 除非括号中的信息对叙事至关重要且无法改写到正文中。

常见需删除的内容：
- 英文人名注释：`吉尔伽美什（Gilgamesh）` → `吉尔伽美什`
- 年份范围：`印度河流域文明（公元前3300年至1300年）` → `印度河流域文明`
- 补充说明：`直立人（Homo erectus）` → `直立人`

### 配音稿开头钩子（Opening Hook）

如果源文本本身已经有开头段落（如"序幕"章节），配音稿的自定义钩子**不能复述原文开头**，否则听众会听到两遍相同内容。

正确做法：
- ✅ 用**提问、悬念、数据冲击**等手法引入，不重复正文内容
- ❌ 把原文开头改写一遍放在最前面（等于是念两遍）

示例对比：
- ❌ 原文："三十亿年前，我们以微小原子的形态漂浮在世界的培养皿中"
- ✅ 钩子："三十亿年前，我们不过是漂浮在原始海洋里的微小原子。六百万年后……为什么偏偏是智人活了下来？"

### 分段与段落

- 段落间用**空行**分隔，TTS 引擎会在此处自然停顿
- 不需要每段都加停顿标签，空行本身就是停顿信号
- 重要转折处才加 `<#2#>` 或 `<#3#>` 标签

## 配音稿口语化规范

从书面文本改编配音稿时，必须做以下处理：

1. **去掉所有括号及内容**：`印度河流域文明（公元前3300年至1300年）` → `印度河流域文明`，TTS读括号会割裂节奏
2. **阿拉伯数字转中文**：`2万名` → `两万名`，`282条` → `两百八十二条`，TTS读数字效果差
3. **去掉列表编号**：`法律1：如果…` → 直接写 `如果…`，不像在念清单
4. **书面语转口语**：`发展出一种收割方法` → `学会了收割`；`旨在为法老提供资源` → `好让法老什么都不缺`
5. **长句拆短**：单句超过60字的拆成两句，TTS一口气读不完会很赶
6. **hook不重复正文**：开头hook是抓注意力的钩子，不要和正文第一段说一样的事

## 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 配音一顿一顿 | 停顿标签加太多 | 移除大部分标签，只保留关键位置 |
| 像不同人在说话 | 声调被调过 | 声调全程保持0 |
| 情绪不对 | 语速/音量参数问题 | 用默认语速1，音量2-3 |
| 台词念错 | TTS引擎问题 | 换个音色试试，或调整文字 |
| **朗读有割裂感** | **括号内容/英文注释** | **删除所有括号及内容，只保留纯中文正文** |
| **开头重复** | **自定义钩子复述了原文** | **钩子用提问/悬念引入，不要改写原文开头** |
| **实际时长和估算差很多** | **TTS语速比预估快** | **录音完成后用 ffprobe 验证每段实际时长，再据此调整视频prompt时长。估算不可靠，以实际为准。** |
