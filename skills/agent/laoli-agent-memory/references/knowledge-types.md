# AgentMemory 知识分类标准

## 记忆类型概述

AgentMemory 支持多种记忆类型，每种类型适用于不同的内容。

## 类型定义

### workflow（工作流程）

**用途**：记录操作步骤、最佳实践、工作流程

**特点**：
- 有明确的步骤顺序
- 可重复执行
- 通常涉及多个工具或系统

**示例**：
```
内容："Docker 部署流程：1) 构建镜像 docker build -t app .；2) 推送镜像 docker push app；3) 部署 docker-compose up -d"
类型：workflow
概念：hermes,运维,部署
```

### fact（事实信息）

**用途**：记录环境配置、工具版本、系统状态等客观事实

**特点**：
- 是客观事实，不是主观判断
- 可能随时间变化
- 通常比较简短

**示例**：
```
内容："当前 Python 版本：3.11.11，Node.js 版本：20.x"
类型：fact
概念：hermes,配置,环境
```

### decision（决策记录）

**用途**：记录技术决策、架构选择、方案对比

**特点**：
- 包含决策原因
- 可能包含备选方案
- 通常比较重要

**示例**：
```
内容："选择 PostgreSQL 而不是 MySQL，原因：1) JSONB 支持更好；2) 扩展性更强；3) 社区活跃"
类型：decision
概念：shared,架构,数据库
```

### bug（问题记录）

**用途**：记录 bug 现象、解决方案、预防措施

**特点**：
- 包含问题现象
- 包含解决方案
- 可能包含预防措施

**示例**：
```
内容："Desktop 更新后崩溃：原因是竞态条件，hermes.exe 正在运行时被替换。解决方案：用 python -m hermes_cli.main update 替代 hermes update"
类型：bug
概念：hermes,bug,Desktop
```

### pattern（设计模式）

**用途**：记录代码模式、架构模式、最佳实践

**特点**：
- 可复用
- 通常有名称
- 可能有示例代码

**示例**：
```
内容："Agent 间通信模式：通过 AgentMemory Signal API 实现点对点消息传递，支持 info/request/response/alert 四种消息类型"
类型：pattern
概念：shared,模式,通信
```

## 类型选择指南

### 选择流程

1. **是否涉及步骤流程？** → workflow
2. **是否是客观事实？** → fact
3. **是否是重要决策？** → decision
4. **是否是问题解决？** → bug
5. **是否是可复用模式？** → pattern
6. **其他情况** → 根据主要内容判断

### 混合内容处理

如果一个记忆包含多种类型的内容：

**方案 A：拆分**
```
# 拆分成多个记忆
记忆1：Docker 部署步骤（workflow）
记忆2：Docker 版本信息（fact）
```

**方案 B：选择主要类型**
```
# 选择最主要的内容类型
主要内容是部署流程 → workflow
次要内容（版本信息）可以省略或简化
```

## 内容质量标准

### 好的记忆

```
✅ 内容具体明确
✅ 包含足够的上下文
✅ 可以直接执行或参考
✅ 标签准确完整
```

### 差的记忆

```
❌ 内容过于简略
❌ 缺少关键信息
❌ 无法直接使用
❌ 标签缺失或错误
```

## 示例对比

### 差的示例

```
内容："Docker 部署有问题"
类型：bug
概念：hermes
```

问题：
- 没有具体现象
- 没有解决方案
- 标签不完整

### 好的示例

```
内容："Docker 部署失败：错误信息 'permission denied'，原因：当前用户不在 docker 组。解决方案：sudo usermod -aG docker $USER，然后重新登录"
类型：bug
概念：hermes,bug,Docker,权限
```

优点：
- 有具体现象
- 有原因分析
- 有解决方案
- 标签完整
