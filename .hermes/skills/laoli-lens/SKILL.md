### 项目目录规范

每个选题创建一个独立项目文件夹，所有产出物按类型存放。项目根目录固定为 `<workspace>/项目`，下文用 `<projects_dir>` 代指。

```
<projects_dir>/[项目名]/
├── 文案/
│   ├── 创意方案.md    ← 阶段A产出（创意研发：诊断→多角度→精修→确认）
│   ├── 创作底本.md    ← 第1+2+3步产出（从创意方案自动生成：选题大纲→视觉指令→分镜脚本）
│   ├── 视频剪辑.md    ← 第8步产出
│   ├── 字幕文稿.md  ← 第9步产出
│   ├── 配乐方案.md   ← 第10步产出
│   └── 发布文稿.md    ← 第12步产出（发布文案）
├── 素材/
│   ├── 图片/          ← 分镜图
│   ├── 封面/          ← 第11步封面图
│   ├── 视频/          ← 生成视频
│   │   ├── 原始/      ← 直接生成
│   │   ├── 变速/      ← 第8步产出（按配音长度变速）
│   │   └── 精修/      ← 第8步产出（变速+去水印）
│   ├── 配音/          ← 第7步配音文件
```

## 执行流程

### 前置准备

### 阶段判定

在执行任何步骤之前，先判断当前项目处于哪个阶段：

1. **检查创意方案** → 如项目目录已存在 `文案/创意方案.md`，说明阶段A已完成，进入阶段B
2. **检查创作底本** → 如项目目录已存在 `文案/创作底本.md` 但无 `创意方案.md`，说明是旧流程项目，直接进入阶段B（从已有步骤继续）
3. **新项目** → 两者都不存在，从阶段A开始

### 方案名获取

确定方案名。按以下优先级获取：

1. **读取项目文件** → 如项目目录已存在 `文案/创作底本.md`，读取其开头的 `> 方案：<方案名>`（断点续做/中间插入时自动恢复）
2. **用户直接提供** → 如用户已告知方案名，直接使用
3. **询问用户** → 上述均未取得时，询问用户指定方案名

确认方案名后，读取 `references/profiles/<方案名>.md` 获取方案设定、视觉风格设定、叙事设定（含叙事类型/亚型选择、风格铁律、节奏模板）和制作规范。

### 可用方案

| 方案名 | 定位 | 适用场景 |
|--------|------|---------|
| `history_outsider` | 历史局外人系列，4亚型轮换 | 传统历史短视频（深沉、悲悯调性） |
| `fate_crossroads` | 命运岔路系列，故事驱动 | 新方向：小人物×大事件×意外因果（猎奇、润物细无声） |
| `dubai-rise` | 迪拜崛起系列，8期规划 | 城市发展类选题（已停更，仅存档参考） |

> ⚠️ **选题库入口**：项目目录 `文档/选题库/INDEX.md` 是选题管理索引，包含所有储备选题的状态。制作视频时应先读取INDEX.md选择待制作选题，再读取对应选题文件。

> ⚠️ **阶段A优先**：新项目必须先执行阶段A（创意研发），产出 `创意方案.md` 后再进入阶段B。跳过阶段A直接做阶段B会导致叙事同质化、视觉单调。

### 执行顺序

### 阶段A执行（创意研发）

执行 `references/steps/A0-creative-rnd.md`，流程：
1. 选题诊断 → 2. 多角度切入 → 3. 🔴 用户选择方向 → 4. 精修方案 → 5. 🔴 用户确认 → 写入 `创意方案.md`

阶段A完成后，自动进入阶段B。

### 阶段B执行（技术执行）

第1步 → 第12步依次执行，每一步的标准动作：

1. **加载步骤文件** → 读取 `references/steps/0N-xxx.md`（如第3步就读 `references/steps/03-script.md`）
2. **执行** → 按步骤文件指引完成本步制作
3. **产出物存放到项目目录** → 按「项目目录规范」写入对应位置
4. **调评审**（步骤文件末尾已定义此硬性条件）→ 必须通过 `delegate_task` 调用评审子 agent。评审指令存储在 `references/reviewer/reviewer.md`，调用时将其内容作为 context 传入，并附上动态路径参数：

   ```python
   delegate_task(
     goal="评审项目「<项目名>」的「<第N步-名称>」产出物",
     context=(
       <references/reviewer/reviewer.md 全文>
       + "\n---\n传入参数：\n"
       + "项目=<项目名> 步骤=<第N步-名称> [方案=<方案名>]\n"
       + f"skill_dir={skill_dir}\n"       # 当前 skill 的绝对路径
       + f"projects_dir={projects_dir}"   # 项目根目录，即`<workspace>/项目` 的绝对路径
     ),
     toolsets=["terminal", "file", "vision"]
   )
   ```

   > ⚠️ **评审员 spec 同步**：评审子 agent 从磁盘加载步骤文件和方案配置。如果用户在本次会话中修改了这些文件（如调整字数上限、修改制作规范），评审员可能加载到旧版本。修复方法：在 context 中显式附上关键约束变更（如"单分镜台词上限：20-80字"），确保评审员使用最新规范。

5. **处理评审结果**，根据步骤角色不同有两种处理路径：

   **🔴 必须确认的步骤**（第3步-分镜脚本、第4步-定妆照、第5步-分镜图、第6步-生成视频）：
   - ✅ 通过 → 告知 Owner 评审通过，**然后立即停止执行，不再继续下一步**。必须等到 Owner 明确说"继续"、"下一步"或类似确认后，才能进入下一步。
   - ⚠️ 需确认 → 暂停，等待 Owner 决策
   - ❌ 不通过 → 按修改意见修复，修复后回到第4步（重新提交评审）

   > ⚠️ **停止点规则**：🔴 步骤评审通过后，Agent 必须立即停止输出后续步骤的内容，只告知 Owner "本步通过，请确认是否继续下一步"。如果 Agent 继续执行下一步，视为违反流程。

    **⏭️ 自动通过的步骤**（其余步骤）：
   - ✅ 通过 → 通知 Owner"本步通过"，**自动进入下一步**
   - ⚠️ 需确认 → 仍需暂停，等待 Owner 决策（⚠️ 表示存在需要人工判断的问题，不能跳过）
   - ❌ 不通过 → 按修改意见修复，修复后回到第4步（重新提交评审）
6. **进入下一步** → 重复 1-6，直到第12步完成

> ⚠️ 注意：每个步骤文件末尾都有 **✅ 本步完成条件** 块，明确要求调评审后才能宣告完成。这是本步的闭环条件，不得跳过。

### 收尾
第12步完成后，汇总最终产出物清单，告知 Owner 管线执行完毕。

### 断点续做 / 中间插入

当对话中断后恢复，或用户要求从中间步骤开始（前序步骤已在外部完成）：

1. **定位步骤** → 读取 `<projects_dir>/[项目名]/` 下的已有产出物，判断已执行到哪一步
2. **恢复方案名** → 按「前置准备」的优先级获取方案名
3. **恢复上下文** → 读取已有产出物（如 `文案/创作底本.md`）获取已完成的选题大纲、角色设定、分镜脚本等，作为后续步骤的输入
4. **继续执行** → 从最近未完成的步骤开始
5. **已完成的步骤无需重做**，除非 Owner 要求修改
