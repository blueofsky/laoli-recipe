# Lessons Learned

## 2026-06-16：阶段A创意研发架构引入

**问题**：laioli-lens 管线的第1-3步（选题大纲→视觉指令→分镜脚本）本质上是"填表"而非"创作"。6段节奏模板强制执行导致所有视频叙事结构同质化，评审员只检查格式合规不评估创意质量，最终产出"质量一般"。

**根因**：
1. 强制节奏模板（6段固定结构）压制了选题本身的叙事特性
2. 评审标准是"格式合规"而非"创意潜力"
3. 没有创意迭代环节——线性流程无法产出多个方案对比
4. 方案配置中"风格铁律"（灵魂）被"节奏模板"（拐杖）压住

**解决方案**：引入阶段A（创意研发），将创意决策从技术执行中分离：
- 阶段A：选题诊断→多角度切入→用户选方向→精修方案→确认
- 阶段B：原第1-12步技术执行（第1步改为从创意方案自动生成）
- 风格铁律升格为"创作核心准则"，节奏模板降为"默认参考"
- 评审标准从"格式合规"改为"创意潜力"（钩子冲击力+意外感+视觉辨识度+金句质量）

**关键设计决策**：
- 阶段A产出 `创意方案.md`，是整个管线的灵魂
- 技术执行阶段读取创意方案自动生成创作底本，不再需要用户手写
- 用户参与点精简为：选题（1句话）→ 选方向（A/B/C）→ 确认方案
- 创意方案评审使用5维度评分（钩子30%+意外感25%+视觉辨识度20%+金句15%+铁律10%）

**适用范围**：此架构适用于所有以"创意质量"为核心竞争力的内容管线，不仅限于历史短视频。

## 2026-06-16：协和号空难创意研发——戏剧性优先于模板

**问题**：阶段A首次实跑（协和号空难），agent初始方案仍然受"60秒时长限制"和"6段模板"思维束缚，产出的方案被用户评价为"不是很饱满""小细节决定成败的故事太多太多了"。

**根因**：
1. Agent下意识按"短视频公式"思考，先定时长再塞内容，而非让故事决定时长
2. 把"蝴蝶效应"简化为"金属片→坠毁"的线性因果链，缺乏戏剧层次
3. 用旁白"告诉"观众发生了什么，而非用画面"让"观众身临其境

**用户的关键纠正**：
- "我没说必须60s啊" → 故事长度由内容决定，不由模板决定
- "只有充满电影和电视的戏剧性才有人看，否则再好的寓意都白扯" → 戏剧性是第一优先级，寓意是副产品
- "要的就是出人意料，动人心神" → 引用《死神来了》作为风格参考
- "如果直接说出来，那太没意思了" → 不要直接告诉观众道理，让他们自己感受

**最终方案的核心手法**：希区柯克式悬念（观众知道桌下有炸弹，角色不知道）。观众从第一秒就知道"会出事"，但只能跟着那天的人一步步走进灾难。不依赖旁白传递信息，而是用画面+沉默+音效让观众身临其境。

**关键教训**：
1. **戏剧性 > 格式合规**：阶段A的评审标准虽然已改为"创意潜力"，但agent在执行时仍会下意识回到"填表模式"。必须在A0步骤文件中明确强调：先想故事，再想格式
2. **不要预设时长**：创意方案阶段不约束时长，让故事自然展开。时长在技术执行阶段根据分镜和配音自然决定
3. **画面自己说话**：最好的分镜不是"旁白告诉你发生了什么"，而是"画面让你身临其境"。沉默段（无旁白）往往比有旁白更有力量
4. **反差是钩子**：最强的钩子不是"信息量大"，而是"反差大"——最强的系统vs最小的漏洞，24年安全vs200秒终结
5. **不要说教**：金句是从故事中自然长出来的，不是外挂的总结。"终结一个时代的，不是更大的飞机，是一块金属片"比"机器没有良心但人有"有力10倍

**适用场景**：所有历史短视频、灾难叙事、人物传记类内容。戏剧性优先原则适用于所有需要"让人停下来听"的内容。

## 2026-06-16：视频时长选择——按戏剧功能选，不按默认值

**问题**：agent在选择视频时长时，总是默认用8s或10s，从不考虑4s或6s。用户指出后反思发现，这是因为agent把时长当作"技术参数"而非"戏剧工具"。

**用户纠正**："你总是选8或者10，其他2个咋不选？"

**正确的时长选择逻辑**：

| 时长 | 戏剧功能 | 适用场景 |
|------|---------|---------|
| **4s** | 冲击瞬间、闪帧、"来不及反应" | 轮胎碾过碎片、爆炸瞬间、关键转折 |
| **6s** | 句号感、静止的力量 | 博物馆凝视、最后一眼、沉默段 |
| **8s** | 标准叙事、有明确起止 | 滑行、飞行、一般动作 |
| **10s** | 沉浸感、情绪铺垫 | 机舱内景、大场景氛围、需要呼吸空间 |

**关键教训**：
- 4s的张力有时比8s更强——"轮胎逼近碎片"从8s改为4s后，分析确认"张力显著更强"
- 最后一个画面用6s而非8s——"句号感"比"省略号"更有力
- 时长是戏剧工具，不是技术约束。在创意方案阶段不预设时长，在视频生成时根据每个场景的戏剧功能选择

## 2026-06-16：MiniMax TTS停顿标签——少即是多

**问题**：agent在MiniMax TTS台词中大量使用`<#N#>`停顿标签，导致配音"一顿一顿的"，非常不自然。

**用户纠正**："加了停顿，听起来一顿一顿的，好不自然啊"

**根因**：MiniMax TTS引擎本身就会在句号、逗号处自然停顿。手动添加`<#N#>`标签会在每个标记处强制插入额外停顿，破坏自然语流。

**正确做法**：
- **默认不加停顿标签**——让TTS引擎自然处理标点处的停顿
- **只在需要"刻意沉默"时使用**——即你想让观众屏住呼吸的那几秒
- 全片通常只用2-3个停顿标签，放在关键转折点

**MiniMax停顿语法**：`<#秒数#>`，如`<#1.5#>` = 停顿1.5秒

**MiniMax情绪/语气词语法**：`(笑声)` `(正常换气)` `(吸气)` 等

**经验法则**：
- 场景开头的长沉默：`<#2.5#>` 或 `<#3#>`（让画面先说话）
- 关键转折前的停顿：`<#1#>` 或 `<#1.5#>`（制造悬念）
- 结尾金句前的停顿：`<#1.5#>`（最后一击的蓄势）
- 其余全部不加——逗号、句号处TTS会自己停

## 2026-06-16：Google Flow共享链接提取视频URL

**问题**：用户通过Google Flow共享链接分享生成的视频，需要agent分析视频质量。

**解决方案**：用浏览器打开Flow共享链接，通过JS提取video元素的src属性获取CDN URL。

**方法**：
```javascript
// 在浏览器console中执行
document.querySelector('video')?.src || document.querySelector('video')?.currentSrc
```

返回的URL格式：`https://flow-content.google/video/{uuid}?Expires=...&KeyName=...&Signature=...`

**注意事项**：
- 共享链接需要登录态才能访问（但浏览器可能已缓存登录态）
- 提取的CDN URL有时效性（Expires参数），过期后需要重新提取
- 也可以直接用共享链接让video_analyze解析（部分情况可行）

**适用场景**：所有通过Google Flow生成并共享的视频素材审查。

## 2026-06-16：Google Flow工作流验证——替代agnes API

**背景**：用户有Google AI Pro会员，可用Google Flow（Nano Banana 2 Pro图片+Gemini Omni Flash视频）替代agnes API。

**验证结果**：
- Nano Banana 2 Pro图片质量优秀，支持在线微调（如修改飞机型号）
- Gemini Omni Flash视频支持4s/6s/8s/10s多时长，比agnes的3-15秒更灵活
- Omni Flash视频可能有两个镜头叠化（开头静态+后面动态），可在线剪辑裁掉
- 共享链接可通过浏览器JS提取CDN URL，用video_analyze分析
- Pro会员额度充足，图片免费，视频消耗credits

**与agnes对比**：
| 维度 | agnes | Google Flow |
|------|-------|-------------|
| 图片费用 | 免费 | 免费（Pro会员） |
| 视频费用 | 免费 | 消耗credits |
| 视频时长 | 3-15秒可调 | 4/6/8/10秒可选 |
| 视频分辨率 | 704px（非标） | 1080p标准 |
| 在线编辑 | 无 | 支持裁剪、微调 |
| 图片微调 | 需重新生成 | 可在图片下输入框调整 |

**结论**：用户有Pro会员时，优先使用Google Flow。agnes作为备用。

## 2026-06-16：MiniMax TTS配音参数——全片保持一致

**问题**：agent为不同场景推荐不同的声调（pitch）参数，导致用户担心"不像一个人了"。

**用户纠正**："声调不能变，要不然不像一个人了。"

**根因**：Agent把情绪变化理解为需要调整所有TTS参数（语速、声调、音量），但实际上：
- **声调（pitch）必须全片一致**——同一个人叙述，不能因为场景不同变声
- **语速（speed）可以微调**——但实测发现默认值1就是对的，不需要刻意放慢
- **音量（volume）基本不变**

**正确做法**：
- 声调：全程0（不变）
- 语速：全程1（默认），不刻意放慢
- 音量：微调（2-3），不要太离谱
- **情绪变化靠文本内容和停顿标签实现，不靠参数调整**

**配音文本量参考**：
- 8秒视频：20-30字配音 + 无声留白
- 10秒视频：30-40字配音 + 无声留白
- 4秒视频：10-15字配音（几乎填满）
- 不是每秒都要有配音——沉默段是叙事的一部分

## 2026-06-16：剪映字幕文稿格式——用于文稿匹配

**问题**：用户需要剪映"文稿匹配"功能用的字幕文稿，但agent最初生成的是配音脚本（带参数和注释），不是纯文本格式。

**正确格式**：
- 纯文本（.txt），每段配音用空行分隔
- 无配音的场景直接跳过（不写"无配音"）
- 按时间顺序排列
- 剪映会自动根据音频时间轴对齐

**示例**：
```
2000年7月25日，巴黎。协和号正在等待起飞...

乘客在看报纸，空姐在走动...

轮胎碾过了一块碎片，悲剧发生了。

机长看见了火...
```

**保存位置**：`项目目录/文案/字幕文稿.txt`

## 2026-06-16：封面图设计——3:4竖版排版方案

**场景**：视频封面需要3:4竖版，带文字标题。

**排版方案（三层结构）**：
| 层级 | 内容 | 位置 | 字号 | 颜色 |
|------|------|------|------|------|
| 主标题 | 核心数字/关键词 | 顶部8%-15% | 最大（96px级） | 白色+橙色强调 |
| 副标题 | 补充说明 | 顶部22% | 中等（22px级） | 白色85%透明度 |
| 底部标语 | 金句/hook | 底部6% | 最小（18px级） | 白色60%透明度 |

**关键原则**：
- 文字不压主体（飞机/人物在中间，文字在顶部和底部）
- 主标题是第一视觉点——观众第一眼看到的就是它
- 底部标语最弱——感兴趣的人会看到，不感兴趣的人不会被干扰
- 在剪映中用文本框逐个添加，手动调整位置
