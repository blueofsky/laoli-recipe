# laoli-reviewer 评审协议 v1

> 协议版本：v1 | 对应 Creator 指南：[review-guide.md](../laoli-shorts/references/review-guide.md) v1

---

## 1. 角色定义

| 角色 | 职责 | 当前执行者 |
|:----:|------|:----------:|
| **Creator**（创作者） | 按管线步骤生成产出物，填写提交说明 | AI Agent（执行 laoli-shorts 管线者） |
| **Reviewer**（评审员） | 按协议评审产出物，更新评审记录 | **本 Agent**（当前执行评审者） |
| **Owner**（决策者） | 通知 Reviewer 接审 → 处理「需决策」事项 → 通知 Creator 查看结论，最终批准 | laoli（项目决策人） |

> **说明**：角色是抽象职能，当前执行者是具体实现。更换工具时只需更新"当前执行者"列，协议主体不变。

## 2. 评审分级

### 🟢 L1 快速通过（文件级）

**适用场景**：方案文档、文案草稿、配置文件等非技术产出

**必做检查项**：

- [ ] 产出文件存在且路径正确
- [ ] 文件格式符合预期（扩展名、编码）
- [ ] 必填字段/段落完整（无空缺）

**输出要求**：1-3行结论即可，无需详细表格

---

### 🟡 L2 标准评审

**适用场景**：图片、视频、音频等媒体产出，以及需要格式/引用规范性检查的文本产出（如视觉指令、分镜脚本）

**必做检查项**：

**媒体类（图片/视频/音频）**：
- [ ] L1 全部检查项
- [ ] 目视抽查（至少1个样本的关键帧/片段）
- [ ] 关键参数在合理范围（分辨率、时长、文件大小）
- [ ] 与上游产出物的一致性（如台词 vs 字幕、prompt vs 实际图）

**文本类（视觉指令/分镜脚本）**：
- [ ] L1 全部检查项
- [ ] 格式规范性：字段/区块结构完整，无格式遗漏
- [ ] 引用完整性：是否引用了对应角色的视觉锚点、风格后缀
- [ ] 内容逻辑性：分镜划分合理、台词与场景匹配、提示词负向排除完整

**输出要求**：简要表格 + 结论，重点标注异常项

---

### 🔴 L3 深度验证（技术级）

**适用场景**：配音、视频剪辑、格式转换等技术敏感步骤

**必做检查项**：

- [ ] L2 全部检查项
- [ ] 技术参数全量提取（ffprobe / 文件属性 / 元数据）
- [ ] 数值交叉比对（如音频时长 vs 视频时长 vs 台词字数）
- [ ] 边界情况检查（最大值、最小值、异常值）
- [ ] 特殊处理验证（去水印、去金句、变速系数、裁剪规格）
- [ ] 跨步骤一致性（与之前步骤的产出物是否矛盾）

**输出要求**：详细数据表 + 问题标注 + 风险评估 + 建议

---

### 步骤默认级别映射

| 步骤类型 | 默认级别 | 典型产出 | 说明 |
|----------|:-------:|----------|------|
| 创作底本（选题大纲） | L1 | `创作底本.md` §1 | 核心冲突完整性、3秒钩子可视觉化、节奏段规划 |
| 创作底本（视觉指令） | **L2** | `创作底本.md` §2 | 角色视觉锚点是否差异化（不克隆脸）、定妆照提示词格式规范、每条需含文化特征和风格后缀 |
| 创作底本（分镜脚本） | **L2** | `创作底本.md` §3 | IMAGE PROMPT 是否引用对应角色视觉锚点、风格后缀是否追加、负向排除完整、台词含感官细节 |
| 定妆照 | L2 | .jpg 图片 | 视觉产出 |
| 分镜图 | L2 | .jpg 图片序列 | 数量多，抽样+全量统计 |
| 生成视频 | L2 | .mp4 视频序列 | 媒体产出 |
| **配音** | **🔴 L3** | **.mp3 音频** | **技术敏感：时长匹配** |
| **视频剪辑** | **🔴 L3** | **.mp4 视频 + .mp3 音效** | **技术敏感：音画对齐 + 音效提取** |
| 字幕文稿（台词提取） | L2 | .txt 文本 | 准确性重要，需逐段对比 |
| 配乐方案 | L1 | .md 方案文档 | 建议性文档 |
| 封面图 | L2 | .png/.jpg 最终图 | 最终视觉产出 |
| 发布文稿 | L1 | .md 文案 | 发布材料 |

> **规则**：Reviewer可根据实际情况**升级**级别（如发现可疑点），但不可**降级**。
> **提示**：创作底本三步（选题大纲 / 视觉指令 / 分镜脚本）虽都写入 `创作底本.md`，但每次提交只有对应 § 有新增内容，Reviewer 仅检查该 § 的产出即可。选题大纲为 L1 快速通过，视觉指令和分镜脚本按 L2 标准评审。

## 3. 状态机

```
                    Creator提交
   ┌─────────────┐ ───────────────→ ┌─────────────┐
   │  ⏳ 待评审   │                   │  🔄 评审中   │
   └─────────────┘                   └──────┬──────┘
       ↑                                   │
       │                                   │
       │  Owner:❌ 不通过                   │  Reviewer完成评审
       │  或 Creator重新提交                │
       │                                   │
       │                       ┌───────────┼───────────┐
       │                       ↓           ↓           ↓
       │                 ┌──────────┐ ┌──────────┐ ┌──────────┐
       │  ←Owner驳回    │ ✅ 通过   │ │ ⚠️ 需确认 │ │ ❌ 不通过 │
       │                 └────┬─────┘ └────┬─────┘ └────┬─────┘
       │                      │            │            │
       │                      │      Owner决策    Creator修改后重交
       │                      │            │            │
       │                      │     ┌──────┴────┐      │
       │                      │     ↓           ↓      │
       │                      │  ┌────────┐ ┌────────┐ │
       │                      │  │ ✅ 通过 │ │ ❌ 不通过 │ │
       │                      │  └───┬────┘ └───┬────┘ │
       │                      ↓      │          └──→───┘
       │                  进入下一步骤
       └──────────────────────┘
```

### 状态定义

| 状态 | 含义 | 触发条件 | 谁可转换至此 |
|------|------|----------|:-----------:|
| ⏳ 待评审 | Creator已提交，等待Reviewer介入 | Creator完成产出并填写提交说明 | **Creator** |
| 🔄 评审中 | Reviewer正在执行检查 | Reviewer开始按协议评审 | **Reviewer** |
| ✅ 通过 | 无问题，或问题已被接受为非阻塞 | Reviewer完成评审且无阻塞问题 | **Reviewer**（自动） |
| ⚠️ 需确认 | 有歧义/风险点，需要Owner拍板 | Reviewer发现无法自行判断的事项 | **Reviewer** |
| ❌ 不通过 | 存在阻塞问题，必须修改后重审 | Reviewer发现严重错误或遗漏 | **Reviewer** |

### 权限矩阵（每个角色在每个状态下可执行的操作）

| 状态 | Creator（创作者） | Reviewer（评审员） | Owner（决策者） |
|------|:----------------:|:------------------:|:--------------:|
| **⏳ 待评审** | 查看自己的提交 | 🔵 **接审** → 进入 🔄 评审中 | 查看 |
| **🔄 评审中** | 查看评审进度 | 🔵 **完成评审** → ✅ 通过/⚠️ 需确认/❌ 不通过 | 查看 |
| **✅ 通过** | 查看结果 | 查看历史记录 | 🔵 **确认** → 进入下一步骤<br>🔵 **可驳回** → ❌ 不通过（事后发现问题的兜底） |
| **⚠️ 需确认** | 查看待决策事项 | 查看等待决策 | 🔵 **决策** → ✅ 通过 或 ❌ 不通过 |
| **❌ 不通过** | 🔵 **修改后重交** → 回到 ⏳ 待评审 | 查看等待重交 | 查看 / 🔵 可强制干预 |

> 图例：
> - 🔵 = 该角色在此状态下的**主动操作**
> - 其他格子 = 该角色在此状态下的**只读权限**
> - Owner 始终拥有**最高干预权**，可在任意状态介入并强制改变状态（但应尽量避免，以维持流程正常运转）

### 状态转换规则

- `⏳ 待评审 → 🔄 评审中`：Reviewer 接审时自动转换
- `🔄 评审中 → ✅ 通过`：Reviewer 评审通过时自动转换
- `🔄 评审中 → ⚠️ 需确认`：Reviewer 遇到触发条件（见第4节）时使用
- `🔄 评审中 → ❌ 不通过`：Reviewer 发现阻塞问题时使用
- `⚠️ 需确认 → ✅ 通过`：Owner 决定接受当前状态
- `⚠️ 需确认 → ❌ 不通过`：Owner 决定不接受，要求修改
- `✅ 通过 → ❌ 不通过`：Owner 事后发现问题时强制驳回（应尽量避免，见权限矩阵说明）
- `❌ 不通过 → ⏳ 待评审`：Creator 修改完成后重新提交

## 4. 【需用户决策】触发条件

### 必须使用此机制的场景（满足任一即触发）

1. **多选项决策**
   - 存在2个以上可行方案，需要Owner选择偏好
   - 例：「封面图A偏叙事感，B偏情绪张力，选哪个？」

2. **资源投入决策**
   - 修改需要额外的时间、预算、API额度
   - 例：「S08水印位置不佳，重新生成消耗XX额度，是否重做？」

3. **风格/创意主观判断**
   - 无法用客观标准衡量的问题
   - 例：「定妆照表情偏紧张还是偏平静更贴合人物？」

4. **跨步影响决策**
   - 当前决策会影响其他已完成的步骤
   - 例：「如果调整配音语速，已完成的视频剪辑需要重做」

### 不应使用此机制的场景（直接在评审结论中以"小建议/非阻塞"形式记录即可）

- 单一方案的微小优化建议
- 信息补充类反馈（如补充标签、加引导语）
- 经验沉淀类警告（供未来项目参考）
- 格式美化类建议

### 使用方式

当触发【需用户决策】时：

```markdown
### 第N步：{任务名}
**状态**：⚠️ 需确认 | **指令**：等待Owner决策

#### Round {n}

...

- **Reviewer**
    ...

    **⚠️ 需Owner决策**：
    - [ ] 决策项A的选项描述
    - [ ] 决策项B的选项描述

    **评审结论**：⚠️ 需确认。等待Owner决策。

- **Owner**
    （Owner在此处回复决策结果）
```

## 6. 评审输出规范

### 结论语句式

统一使用以下句式，确保语义清晰无歧义：

| 场景 | 句式模板 | 示例 |
|------|---------|------|
| 通过 | ✅ 通过。{正面总结} | ✅ 通过。10段音频全部达标。 |
| 需确认 | ⚠️ 需确认。{问题描述}+{可选方案} | ⚠️ 需确认。音频超长120%，如何处理？ |
| 不通过 | ❌ 不通过。{阻塞原因}+{修改方向} | ❌ 不通过。S03图片分辨率不足，需重新生成。 |

### 问题标注规范

在评审过程中发现问题时的标注方式：

| 严重度 | 标记 | 含义 | 要求 |
|--------|:----:|------|------|
| 致命 | 🔴 | 阻塞性问题，必须修复才能继续 | 必须进入 ❌ 不通过 状态 |
| 严重 | 🟠 | 重要缺陷，强烈建议修复 | 建议进入 ⚠️ 需确认 或 ❌ 不通过 |
| 一般 | 🟡 | 改进建议，不影响流程 | 记录在结论中作为"小建议" |
| 提示 | 🔵 | 信息补充，纯参考 | 可选记录 |

### 数据表规范

当需要进行数值对比时，使用统一表格格式：

```markdown
| 对比项 | 期望值 | 实际值 | 偏差 | 结果 |
|--------|:-----:|:-----:|:---:|:----:|
| {名称} | {标准} | {实测} | {差异} | ✅ 通过/❌ 不通过 |
```
